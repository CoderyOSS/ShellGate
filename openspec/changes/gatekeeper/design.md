## Context

AI agents run inside a sandbox container on a VPS. Currently, `gh` CLI and `git` have direct GitHub access via a GitHub App token generated by `github-app-token` and injected via `gh auth login` or `GH_TOKEN` env var. There is no human oversight — agents can push to any branch, create/delete repos, merge PRs, and trigger workflows without restriction.

The sandbox container shares a host with other services managed by the Willow infrastructure system. The host runs Caddy, Tailscale, supervisord, and a blue/green deployment orchestrator.

## Goals / Non-Goals

**Goals:**
- Every GitHub-mutating action requires explicit human approval before execution
- Agent experience is transparent — `gh` and `git` commands work identically from the agent's perspective
- Approval workflow accessible on-the-go via Telegram and a web dashboard
- Full audit trail of every command that touches GitHub
- Pre-approval workflow so agents can request permissions in advance for planned work
- Extensible to gate non-GitHub commands in the future (the shell interception layer is general-purpose)
- Gate-server runs on the host, credentials never enter the container

**Non-Goals:**
- Per-agent identity or trust levels (all agents treated equally for v1)
- Fine-grained ref-level permissions (e.g., allow push to feature branches but not main) — deferred to v2
- Multi-user support — single human approver
- Proxying individual HTTP API calls (we intercept at the shell command level, not the network level)
- Bash-specific support (zsh with bash symlink covers the container use case)

## Decisions

### 1. Shell-level interception via patched zsh, not TLS proxy

**Decision**: Patch zsh to add a pre-exec hook that checks every external command against gate-server before fork+execve. Symlink `/bin/bash` → `/bin/zsh` so all tools hit the patched shell.

**Alternatives considered**:
- TLS MITM proxy (like mitmproxy): Intercepts at the HTTP level. Would catch raw API calls, but requires CA cert management, HTTPS_PROXY configuration, and parses raw API paths instead of human-readable commands. Much more complex for no additional security benefit when the container has no credentials anyway.
- Bash DEBUG trap with extdebug: Can suppress commands but has exit code propagation issues — returning non-zero to skip the command makes the caller see exit 1 even when the gated command succeeded. No clean way to both skip and return the correct exit code.
- Function overrides in bashrc: `gh() { gate-exec "$@"; }` — bypassable with `command gh`, `/usr/bin/gh`, PATH manipulation. Mitigated by no-creds-in-container, but still whack-a-mole for new commands.
- LD_PRELOAD hooking execve(): Works with any shell, but bypassable (unset LD_PRELOAD, static binaries). Doesn't give us the clean env-injection story (token injected after fork, before execve, only in child process memory).

**Rationale**: The patch hooks zsh's `execcmd()` in `Src/exec.c` — the internal function zsh calls before fork+execve for every external command it directly invokes. This means every command typed at the prompt or run via zsh's own execution path goes through gate-checking. Commands invoked BY child processes (e.g., `make` running `git push` internally) are NOT intercepted — but they have no credentials to use anyway. The security model is credential-absence; the zsh patch adds approval UX for direct commands. The hook fires before fork, so rejected commands never become processes. Env injection happens after fork but before execve, so credentials only exist in the child's memory and evaporate on process exit.

### 2. zsh patch now, upstream PR + module later

**Decision**: Ship with a small patch (~20 lines in `Src/exec.c`) applied to zsh releases. Submit a PR to zsh to add a `pre_exec` hookdef to the module API. Once merged, ship a pure zsh module instead of a fork.

**Rationale**: zsh's module API has `WRAPDEF` for wrapping shell functions but no hook for external command execution. The execution path `execcmd() → execute() → zexecve() → execve()` has no callback infrastructure. A PR adding a `pre_exec` hookdef would let any module register for pre-exec interception. The patch is the bridge until upstream accepts.

### 3. Unified permission model — all permissions are toggles with TTL

**Decision**: Every permission (repo:read, pr:create, git:push, etc.) is a toggle with an ON/OFF state and a TTL. Reads default to ON with indefinite TTL. Writes default to OFF. Pre-approvals turn OFF → ON with a configurable TTL. One code path for all checks — no separate auto-approve vs gate logic.

**Alternatives considered**:
- Separate auto-approve list + gate list: Two code paths, two configuration formats, harder to reason about. The unified model is simpler and equally expressive.

**Rationale**: Configuration drives behavior. User can tighten reads (flip repo:read to OFF) or loosen writes (pre-approve pr:create for 2 hours). Same `is_permission_active(action, repo)?` check everywhere.

### 4. Commands execute in the container, gate-server provides tokens

**Decision**: The patched zsh runs commands locally in the container. Gate-server returns credentials (GH_TOKEN) to inject into the child process environment. Gate-server does NOT proxy command execution for most commands — it authorizes and provides tokens.

**Exception for `gh`**: `gh` CLI is not installed in the container. When the agent runs `gh` commands, the patched zsh sends the full command to gate-server which executes `gh.real` on the host and streams stdout/stderr back. This is because `gh` requires authentication configuration that we don't want in the container at all.

**Limitation**: Proxy mode has no TTY. gh commands requiring interactive input (`gh pr create`, `gh issue create`, etc.) will fail or enter non-interactive mode. Agents should pass `--fill` / `--body` / `--title` flags for non-interactive use. Interactive gh commands are not supported in v1.

**Signal forwarding**: SIGINT (Ctrl+C) from the container is forwarded through the socket to the gh process on the host. Exit codes from gh are propagated back to the patched zsh.

**Rationale**: Running commands locally means git handles its own IO (interactive merge conflicts, progress bars, large output streams) without proxying complexity. Token injection is simpler than IO proxying. For `gh`, proxying is acceptable because gh output is typically small and non-interactive (interactive gh commands are not supported in v1).

### 5. SQLite on host for all state

**Decision**: Single SQLite database at `/opt/gate/db.sqlite` stores default_permissions, grants (pre-approvals), approval_requests, audit_log, and config. No external database dependency.

**Rationale**: Single-user system, single host. SQLite is fast, reliable, zero-ops. Audit log rows have a `ttl_until` column — a background task cleans expired rows.

### 6. Two execution modes: proxy and token

| Mode | When | Flow |
|------|------|------|
| **proxy** | `gh` commands (no real gh in container) | gate-server executes command on host, streams output back to patched zsh |
| **token** | `git push/fetch/pull/clone` | gate-server returns short-lived token, patched zsh injects into child process env, command runs in container |
| **proxy** | `curl` to github.com | gate-server rewrites curl command adding `-H "Authorization: Bearer <token>"` and executes on host (deferred to v2; v1 use `gh api` instead) |

### 7. Tech stack

| Component | Technology | Reason |
|-----------|-----------|--------|
| gate-server | Rust (tokio, axum, rusqlite) | Performance, reliability, user preference |
| Patched zsh | C (zsh source + patch) | Only way to hook pre-exec at the right level |
| Web UI | Astro + React (TypeScript), static build | User preference, served by gate-server |
| Telegram bot | Rust (teloxide or similar) | Integrated into gate-server binary |
| MCP server | Rust (same binary, subcommand) | Integrated into gate-server binary |

## Risks / Trade-offs

**[Patch maintenance burden]** → zsh releases are infrequent (~yearly). The patch is ~20 lines against `Src/exec.c`. Rebase is trivial. Upstream PR eliminates this entirely once merged.

**[Token briefly in container process env]** → Token is injected after fork, only in child process memory, evaporates on exec completion. Even if leaked, it's a short-lived GitHub App token (~1h TTL). No persistent credentials in container.

**[Unix socket is a single point of failure]** → If gate-server crashes, ALL commands in the container that require checking hang or fail. Mitigate: gate-server runs under supervisord with auto-restart. Local-only commands (git add, ls, etc.) can be fast-pathed without socket round-trip.

**[preexec fires once per command line, not per command]** → `git push && curl github.com` is one command line. The patched zsh hook fires for each individual execve (each simple command in the pipeline), not once for the whole line. This is correct because we hook at the execcmd level, not the preexec level.

**[Agent could compile and exec a binary directly]** → The compiled binary inherits no credentials. Without GH_TOKEN in its environment, it can't authenticate to GitHub. The gate only controls credential injection, not process creation.

**[Telegram as sole notification channel]** → If Telegram is down, approvals still work via web dashboard. Notification is best-effort. Could add more channels (Discord, Slack) in v2.

## Migration Plan

1. Build gate-server on host, run under supervisord, verify socket connectivity
2. Build patched zsh, test in isolated environment
3. Deploy to sandbox container: install patched zsh, symlink bash, remove credentials, mount socket
4. Verify agent workflow: `gh pr list`, `git add`, `git commit`, `gh pr create` (should trigger approval)
5. Configure Telegram bot, test approve/reject flow
6. Configure default permissions to match current behavior (reads auto-approved)
7. Update Willow Dockerfile.sandbox and service definitions
8. Blue/green deploy with rollback capability

**Rollback**: Restore original bash, restore credentials, remove socket mount. Agent returns to direct GitHub access.

## Open Questions

- Should the zsh patch support caching allowed commands locally (avoid socket round-trip for every command) or always round-trip? Unix socket is sub-ms latency, but scripts with many commands could accumulate delay.
- [RESOLVED] Pending approval request TTL: 5 minutes (configurable in gate-server TOML). After expiry, the request transitions to 'expired' state and the patched zsh receives a 'reject' response with reason 'approval request expired'. Agent sees exit code 77 (custom code for 'gated - expired').
- Should the gate-server expose metrics (Prometheus) for monitoring approval latency, rejection rates, etc.?
- Should the zsh upstream PR include the ability for the hook to modify argv (not just env)? This would enable command rewriting (e.g., redirecting `git push` to a different remote).
